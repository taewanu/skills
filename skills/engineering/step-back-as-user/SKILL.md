# Step Back As User

A perspective-switching skill. Drops the implementation frame and re-enters as the person who actually uses the thing. Surfaces friction without proposing fixes.

## What to do

Drop the current solution. Identify who actually uses this — one line: role, context, what they're trying to get done. Then walk through what they see, touch, and feel at the relevant moment. Surface what's unclear, missing, frustrating, or wrong for them — not for the system.

One concrete user at a time. If the user is already obvious from context, skip naming them and go straight to the experience.

Stop at diagnosis. Don't redesign — the point is to notice friction, not solve it. If the user wants fixes, they'll ask.

## What this is not

- Not a UX audit. One user, one moment, real friction.
- Not a redesign session. Surfacing problems ≠ solving them.
- Not "what could go wrong" brainstorming. Stay grounded in what a specific person actually does.
- Not roleplay. Don't invent persona drama. The user is a tool for noticing, not a character.

## When to skip

- The user already named the friction — go fix it, don't re-discover it.
- Purely internal work (build tooling, refactors with no UX surface) — there's no user moment to step into.
- Exploratory brainstorming where a critical pass would derail momentum.
